
# Paper summary

## E-step
During the E-step, the composition model searches for the best parse tree and composes representations through a deep inside-outside auto-encoder (DIORA).

### Inside
![equation](https://latex.codecogs.com/svg.image?\mathbf{i}_{i,j}) is a weighted average over all possible pairs of constituents:


![equation](https://latex.codecogs.com/svg.image?\mathbf{i}_{i,j}=\sum_{k=i}^{j-1}\hat{w}^k_{i,j}\bar{\mathbf{i}}^k_{i,j})

![equation](https://latex.codecogs.com/svg.image?\hat{w}^k_{i,j}=\frac{\exp(\bar{a}^k_{i,j})}{\sum_{k'=i}^{j-1}\exp(\bar{a}^{k'}_{i,j})})


![equation](https://latex.codecogs.com/svg.image?\bar{\mathbf{i}}^k_{i,j}=f_{\alpha}(\mathbf{i}_{i,k},\mathbf{i}_{k+1,j}))

![equation](https://latex.codecogs.com/svg.image?f_{\alpha}:=\text{Transformers})

![equation](https://latex.codecogs.com/svg.image?\bar{a}^k_{i,j}=\phi_{\alpha}(\mathbf{i}_{i,k},\mathbf{i}_{k+1,j}))

![equation](https://latex.codecogs.com/svg.image?\phi_{\alpha}(\mathbf{l},\mathbf{r})=\frac{\mathrm{MLP}_{\alpha}^{l}(\mathbf{l})^{T}\mathrm{MLP}_{\alpha}^{r}(\mathbf{r})}{\sqrt{d}})

![equation](https://latex.codecogs.com/svg.image?\mathrm{MLP}_{\alpha}(.):=?)

### Outside

![equation](https://latex.codecogs.com/svg.image?\mathbf{o}_{i,j}=\sum_{k\in\mathcal{P}_{i,j}}\check{w}^k_{i,j}\bar{\mathbf{o}}^k_{i,j}), where ![equation](https://latex.codecogs.com/svg.image?\mathcal{P}) is valid immediate parent spans.


![equation](https://latex.codecogs.com/svg.image?\bar{b}^k_{i,j}=\begin{cases}\phi_{\beta}(o_{i,k},i_{j+1,k})\quad\text{if}k>j,\newline\phi_{\beta}(o_{k,j},i_{k,i-1})\quad\text{if}k<{}i.\end{cases})

![equation](https://latex.codecogs.com/svg.image?\check{w}^k_{i,j}:=\text{softMax}(\bar{b}^k_{i,j}))

![equation](https://latex.codecogs.com/svg.image?\phi_{\beta}(\mathbf{p},\mathbf{l})=\frac{\mathrm{MLP}_{\beta}^{p}(\mathbf{p})^{T}\mathrm{MLP}_{\beta}^{l}(\mathbf{l})}{\sqrt{d}})

![equation](https://latex.codecogs.com/svg.image?\phi_{\beta}(\mathbf{p},\mathbf{r})=\frac{\mathrm{MLP}_{\beta}^{p}(\mathbf{p})^{T}\mathrm{MLP}_{\beta}^{r}(\mathbf{r})}{\sqrt{d}})


![equation](https://latex.codecogs.com/svg.image?\bar{\mathbf{o}}^k_{i,j}=\begin{cases}f_{\beta}(\mathbf{o}_{i,k},\mathbf{i}_{j+1,k})\quad\text{if}k>j,\newline{}f_{\beta}(\mathbf{o}_{k,j},\mathbf{i}_{k,i-1})\quad\text{if}k<{}i.\end{cases})




### Loss

# `GenerativeR2D2` — Code Walkthrough

Mapping the code in `model/generative_r2d2.py` directly to the GPST paper's EM framework.

---

## The Two Models as Class Members

```python
self.r2d2 = r2d2      # ← composition model (E-step lives here)
self.gpt  = gpt       # ← generative model  (M-step lives here)
```

These are the two components described in the paper: the **composition model** that induces
parse trees via inside-outside, and the **generative model** that does left-to-right language
modeling conditioned on the induced tree.

---

## `forward()` — The Full EM Step

### E-step — Tree Induction + Representation Computation

This single call does everything in the E-step:

```python
ctx, outside_tgt, ldr_repr, position_ids, tgt_ids, ... = \
    self.r2d2(r2d2_input_ids, chunk_masks, input_ids, masks, r2d2_embeddings, ...)
```

Internally it:
- Runs the **inside pass** → scores all split points → picks the best tree
- Runs the **outside pass** → computes `ctx` (outside representations of every span)
- Returns `ldr_repr` — the **inside span representations**, which are the *representation surrogates* from the paper (Figure 2b)

---

### M-step — Two Losses Computed Jointly

#### Bi-directional loss $\mathcal{L}_{ae}$ — composition model supervision

```python
parser_loss         = self.r2d2.parser_loss(ctx)
outside_embeddings  = self.r2d2.outside_embeddings(ctx)
outside_logits      = self.classifier(self.insideoutside_dense(outside_embeddings))
lm_loss             = F.cross_entropy(outside_logits, outside_tgt)
```

Each token is predicted from its **outside representation** — context from *both sides* of the
token. This is $\mathcal{L}_{ae}$: the auto-encoding loss that corrects the asymmetric feedback
problem and prevents branching bias in the induced trees.

---

#### Uni-directional loss $\mathcal{L}_{ar}$ — generative model supervision

```python
gpt_input = self.up_scale(ldr_repr)                            # ← representation surrogate injected here
gpt_input.scatter_(1, token_indices, input_embeddings)         # actual token positions get real embeddings
cat_input = torch.cat([bos_emb.unsqueeze(1), gpt_input], dim=1)
outputs   = self.gpt(inputs_embeds=cat_input, position_ids=position_ids)
logits    = self.classifier(self.dense(hidden_states))
loss      = F.cross_entropy(logits.permute(0, 2, 1).float(), tgt_ids, ignore_index=-1)
```

This is $\mathcal{L}_{ar}$: autoregressive generation conditioned on pre-computed inside span
representations. Because `ldr_repr` is already computed in the E-step, the entire M-step runs
**in parallel** — no sequential composition dependencies.

---

## The Representation Surrogate in One Line

```python
gpt_input = self.up_scale(ldr_repr)                         # inside repr from E-step → fed into GPT
gpt_input.scatter_(1, token_indices, input_embeddings)      # but actual token positions get real embeddings
```

This is exactly **Figure 2(b)(ii)** from the paper. Pre-computed inside representations replace
the sequential composition inputs, breaking the data dependency chain and enabling full
Transformer parallelism in the M-step.

---

## The Final Loss Split

```python
return R2D2GenOutput(
    struct_loss     = lm_loss + l_height,       # Lae  — shapes the tree structure
    non_struct_loss = 2 * loss + parser_loss,   # Lar  — shapes generation
    ...
)
```

Back in `ddp_trainer_nosync.py`:

```python
scaler.scale(result.struct_loss).backward(retain_graph=True)   # 1st backward — tree
scaler.scale(result.non_struct_loss).backward()                # 2nd backward — generation
```

`retain_graph=True` keeps the computation graph alive between the two backward passes.
This is also where the **gradient stop** from the paper is enforced — `struct_loss` gradients
flow into the split scores, but `non_struct_loss` gradients are blocked from doing so,
preventing the left-branching bias.

---

## Summary: Code ↔ Paper Mapping

| Paper concept | Code location |
|---|---|
| E-step (inside-outside) | `self.r2d2(...)` call in `forward()` |
| Outside representations | `ctx` returned from `self.r2d2` |
| $\mathcal{L}_{ae}$ (auto-encoding loss) | `lm_loss = F.cross_entropy(outside_logits, outside_tgt)` |
| Representation surrogate | `self.up_scale(ldr_repr)` + `scatter_()` |
| $\mathcal{L}_{ar}$ (auto-regression loss) | `loss = F.cross_entropy(logits..., tgt_ids)` |
| Gradient stop on split scores | `retain_graph=True` + separate `.backward()` calls |
| Two-component joint training | `struct_loss` and `non_struct_loss` returned separately |



# `r2d2_insideoutside.py` — Code Walkthrough

This is the heart of the E-step. It implements the **pruned deep inside-outside algorithm**
that induces the parse tree and computes span representations — all without gold trees.

---

## Class Structure

```python
class InsideOutsideModule(R2D2Base):
    self.parser       # TransformerParser  — scores split points
    self.inside_enc   # InsideEncoder      — computes inside span representations bottom-up
    self.outside_enc  # OutsideEncoder     — computes outside span representations top-down
```

The `TensorCache` is a flat key-value store that holds 4 values per span `(i,j)`:

| Slot ID | Name | What it stores |
|---|---|---|
| `e_ij_id` (0) | `e_ij` | inside span representation $i_{i,j}$ |
| `score_sum_id` (1) | `log_p_ij` | accumulated split score (soft-weighted) |
| `score_ijk` (2) | `scores_ijk` | raw split-point scores $\bar{a}^k_{i,j}$ |
| `height_ij` (3) | `h_ij` | tree height (used for height regularization) |

---

## `forward()` — The Full E-step Pipeline

### Step 1 — Score all split points

```python
split_indices, split_scores = self.parser(chunk_input_ids, chunk_masks, ...)
```

`TransformerParser` runs a Transformer encoder over the input and scores every possible
split point $k$ for every span $(i,j)$. Returns:
- `split_indices` — ranked split points (used to prune the chart)
- `split_scores` — raw scores (used later in `parser_loss`)

---

### Step 2 — Initialize the tensor cache with token embeddings

```python
inside_cache = self.create_tensor_cache(seq_lens_np)
inside_cache.scatter(input_cache_ids, [self.e_ij_id], [flatten_r2d2_emb])
```

Each leaf node (token) gets its embedding $e_i$ placed into slot `e_ij_id`.
All other slots start at zero. This seeds the bottom-up inside pass.

---

### Step 3 — Build the chart groups (C++ backend)

```python
tables = CPPChartTableManager(seq_lens_np, self.window_size, split_indices, ...)
target_cache_ids, span_ids, cache_ids, detach_cache_ids = tables.construct_inside_groups(self.device)
```

This is the **pruning** step. Instead of computing all $O(n^3)$ span-split combinations,
the C++ backend uses the ranked `split_indices` to build batched groups of spans that
can be computed in parallel. Each group contains spans at the same "level" of the tree
with no data dependencies between them — enabling the logarithmic parallel time complexity
claimed in the paper.

---

### Step 4 — Inside pass (bottom-up)

```python
score_orders = self.inside(inside_cache, span_embeds, temperature,
                            zip(target_cache_ids, span_ids, cache_ids, detach_cache_ids))
```

See `inside()` section below. After this call, `inside_cache` holds $i_{i,j}$ for every span.

---

### Step 5 — Prepare generation inputs (representation surrogate)

```python
span_masks, split_targets, ldr_cache_ids, position_ids, tgt_ids, token_indices, ext_ids = \
    tables.prepare_generation(score_orders, ...)
```

The C++ backend extracts the **best tree** by greedily picking the highest-scoring split
point top-down from the root. It then produces:
- `ldr_cache_ids` — cache IDs for all nodes in the induced tree in post-order (left-then-right)
- `position_ids` — position IDs for the generative model
- `tgt_ids` — target token IDs for the autoregressive loss $\mathcal{L}_{ar}$
- `split_targets` — gold split points for `parser_loss`

---

### Step 6 — Gather the representation surrogates

```python
ldr_repr = inside_cache.gather(ldr_cache_ids.flatten(), [self.e_ij_id])[0]
ldr_repr = ldr_repr.view(*ldr_cache_ids.shape, -1)  # (N, L, dim)
```

These are the **inside span representations** $i_{i,j}$ for every node in the induced tree,
gathered in post-order. This is exactly the representation surrogate from Figure 2(b)(ii)
of the paper — fed directly into the generative model in the M-step.

---

### Step 7 — Height regularization

```python
inside_height = inside_cache.gather(root_ids, [self.height_ij])[0]
inside_height = torch.where(inside_height > self.height_threshold, inside_height - self.height_threshold, 0)
l_height = (inside_height / height_norm).mean()
```

This penalizes trees that are too flat (small height). Added to `struct_loss` in
`GenerativeR2D2` as `lm_loss + l_height` — it encourages the model to build deeper,
more linguistically plausible trees.

---

### Step 8 — Package context for outside pass

```python
ctx = InsideOutsideContext(
    scores=split_scores,
    split_masks=span_masks,
    split_points=split_targets,
    inside_cache=inside_cache,
    outside_groups=zip(reversed(target_cache_ids), ...)  # ← reversed = top-down
)
```

`ctx` carries everything needed for the outside pass and the two loss computations.
Note `reversed()` — the inside pass goes bottom-up, the outside pass reuses the same
groups but traverses them top-down.

---

## `inside()` — Bottom-up Span Representation

Called once per level of the chart, iterating over `inside_groups`:

```python
e_ij, scores_ij_sum, h_ij = self.prepare_composition(cache_ids, detach_cache_ids, inside_cache)
scores_ijk, c_ijk = self.inside_enc(e_ij)
```

For each candidate split $k$ of span $(i,j)$, the inside encoder combines the left child
$i_{i,k}$ and right child $i_{k+1,j}$ representations and scores the combination.

```python
a_ij = F.softmax(scores_ijk_sum / temperature, dim=-1)  # soft weights over split points
c_ij = WeightedSumFunc.apply(a_ij, c_ijk)               # weighted sum: $i_{i,j} = \sum_k \hat{w}^k_{i,j} \bar{i}^k_{i,j}$
c_ij = self.norm(c_ij)
```

This is the soft inside representation from the paper:

$$i_{i,j} = \sum_{k=i}^{j-1} \hat{w}^k_{i,j} \cdot \bar{i}^k_{i,j}$$

The `temperature` parameter controls the sharpness of the distribution — lower temperature
pushes toward hard argmax (harder tree selection).

```python
h_ij = torch.einsum("ij, ij->i", a_ij, h_ij_next)  # soft-weighted height
inside_cache.scatter(target_cache_ids, [self.e_ij_id, self.score_sum_id, self.height_ij],
                     [c_ij, scores_ij_sum, h_ij])
```

Results are written back to `inside_cache` for the next level up.

---

## `outside()` — Top-down Outside Representations

Initializes the root with a learned embedding:

```python
outside_cache.scatter(root_ids, [topdown_e_ij_slot, ...], [root_embedding, ...])
```

Then iterates top-down through the same groups (reversed), computing for each child span
its outside representation — context from everything *outside* that span:

```python
out_scores, out_ikj = self.outside_enc(parent_ij, child_ikj, parent_ij_score, child_scores)
```

The update uses a **logsumexp-based soft merge** to accumulate outside information from
multiple possible parent contexts:

```python
left_k_sum_scores  = torch.stack([log_ksum_score, out_scores[:, :, 0]], dim=2)
left_k_weights     = F.softmax(left_k_sum_scores, dim=2)
left_weighted_e_ij = left_k_weights[:, :, 0] * weighted_e_ij + \
                     left_k_weights[:, :, 1] * out_ikj[:, :, 0, :]
```

This merges the accumulated outside representation with the new contribution from the
current parent, weighted by their relative scores. The same is done for both left and
right children.

---

## `parser_loss()` — Supervising the Split Point Scorer

```python
scores.masked_fill_(split_masks == 0, float('-inf'))
return F.cross_entropy(scores.transpose(1, 2).float(), split_points[:, :L], ignore_index=-1)
```

Treats tree induction as a classification problem: for each position in the sentence,
predict which split point the model chose. `split_targets` from `prepare_generation()`
serve as pseudo-labels derived from the best tree found during the inside pass.
This is the **self-supervised** part — the model bootstraps its own supervision signal.

---

## Summary: Full E-step Data Flow

```
token embeddings
      ↓
TransformerParser → split_indices, split_scores
      ↓
CPPChartTableManager → batched inside_groups (pruned chart)
      ↓
inside() [bottom-up, parallel levels]
  for each level:
    InsideEncoder(left_child, right_child) → scores_ijk, c_ijk
    softmax(scores_ijk) → a_ij (soft split weights)
    weighted_sum(a_ij, c_ijk) → i_{i,j}  [stored in inside_cache]
      ↓
prepare_generation() → best tree (greedy top-down), ldr_cache_ids, tgt_ids
      ↓
inside_cache.gather(ldr_cache_ids) → ldr_repr  [representation surrogate → M-step]
      ↓
InsideOutsideContext (ctx) → outside pass + loss computation in M-step
```

---

## Key Design Choices and Why

| Choice | Reason |
|---|---|
| Soft weighted sum for $i_{i,j}$ | Differentiable — gradients flow through split weights during training |
| Hard best tree for `ldr_repr` | Breaks data dependency for parallel M-step; paper shows minimal accuracy impact |
| `detach_cache_ids` separate from `cache_ids` | Implements the gradient stop — scores receive gradients only from $\mathcal{L}_{ae}$, not $\mathcal{L}_{ar}$ |
| `reversed()` for outside groups | Reuses inside group structure top-down with zero extra computation |
| Height regularization `l_height` | Prevents degenerate flat trees without any linguistic supervision |
| C++ backend for chart groups | The $O(n^2)$ chart is too slow in pure Python; C++ handles grouping and tree extraction |
