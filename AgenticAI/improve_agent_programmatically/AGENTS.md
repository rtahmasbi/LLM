## §2 Architecture Map

```
fastapi/
├── applications.py       # FastAPI app class — entry point, ties everything together
├── routing.py            # APIRouter + route registration; most routing logic lives here
├── params.py             # Query, Path, Header, Cookie, Body, Form, File param definitions
├── dependencies/
│   ├── utils.py          # Dependency resolution graph, concurrency handling
│   └── models.py         # Internal dependency models
├── security/
│   ├── oauth2.py         # OAuth2 scheme implementations  ⚠️ sensitive — test thoroughly
│   ├── http.py           # HTTP Basic/Bearer auth
│   └── api_key.py        # API key schemes
├── encoders.py           # jsonable_encoder — custom JSON serialization
├── exception_handlers.py # Default handlers for RequestValidationError, HTTPException
├── middleware/           # Middleware utilities (CORS, GZip, etc.)
├── testclient.py         # Starlette TestClient re-export (stable, low churn)
├── openapi/
│   ├── utils.py          # OpenAPI schema generation
│   └── docs.py           # Swagger UI + ReDoc serving
└── _compat.py            # Pydantic v1/v2 compatibility shims  ⚠️ fragile, avoid editing
```

**Key dependency relationships:**
- `applications.py` → `routing.py` → `dependencies/utils.py`
- Route decoration (`@app.get`) is syntactic sugar over `router.add_api_route()`
- `_compat.py` is the abstraction boundary for all Pydantic version differences — go through it, never import from pydantic directly in new code

## §3 Code Conventions

### Type Annotations

```python
# ✅ Use Annotated for dependency injection (current style)
from typing import Annotated
from fastapi import Depends, Query

async def get_items(
    skip: Annotated[int, Query(ge=0)] = 0,
    db: Annotated[Session, Depends(get_db)] = None,
):
    ...

# ❌ Old style — still works but don't add new code this way
async def get_items(skip: int = Query(0, ge=0), db: Session = Depends(get_db)):
    ...
```

### Pydantic Models

```python
# ✅ Pydantic v2 style (current)
from pydantic import BaseModel, model_validator, field_validator

class Item(BaseModel):
    model_config = {"extra": "forbid"}   # use model_config, not class Config

    @field_validator("name")             # use @field_validator, not @validator
    @classmethod
    def name_must_not_be_empty(cls, v):
        ...

# ❌ Pydantic v1 style — do not use
class Item(BaseModel):
    class Config:
        extra = "forbid"
```

### Routers & App Structure

```python
# ✅ Define routers in separate files, include in app
# routers/items.py
router = APIRouter(prefix="/items", tags=["items"])

# main.py
app.include_router(items.router)

# ❌ Never call app.include_router() inside a function or route handler
# ❌ Never define routes inline in main.py for non-trivial apps
```

### Exception Handling

```python
# ✅ Use HTTPException for client errors
from fastapi import HTTPException
raise HTTPException(status_code=404, detail="Item not found")

# ✅ Use request.app.state for app-level state
# ❌ Don't return raw Response objects for error cases — breaks OpenAPI schema
# ❌ Don't catch Exception broadly in route handlers without re-raising or logging
```

### Async vs Sync

- Use `async def` for route handlers that call async I/O (DB, HTTP clients)
- Use plain `def` for CPU-bound or sync-only handlers — FastAPI runs them in a thread pool
- Never call `asyncio.run()` inside a route handler
- Never mix sync and async in the same dependency chain without care — can deadlock

## §4 PR & Contribution Rules

### Branch Naming

```
fix/<issue-number>-short-description     # bug fixes
feat/<issue-number>-short-description    # new features
docs/short-description                   # documentation only
refactor/short-description               # no behavior change
```

### Commit Style

- Use conventional commits: `fix:`, `feat:`, `docs:`, `refactor:`, `test:`, `chore:`
- Keep subject line under 72 characters
- Reference issue numbers: `fix: handle None in jsonable_encoder (#1234)`

### PR Requirements

- Tests required for all changes to public API surface
- Type annotations required on all new public functions
- Docstrings required for new public classes and functions
- Squash merge only — no merge commits in main branch history
- One approval required from a maintainer before merge
- CI must be green: pytest + ruff + mypy all passing

### What Not to Submit

- Don't submit PRs that only add `Optional[X]` → `X | None` type alias changes (will be closed)
- Don't modify `_compat.py` without explicitly discussing Pydantic v1/v2 implications in the PR description
- Don't add new dependencies to core `fastapi/` without prior issue discussion

## §5 Anti-Patterns & Gotchas

These are the things agents and new contributors most commonly get wrong.

### Routing

```python
# ❌ Path parameter name must match function parameter name exactly
@app.get("/items/{item_id}")
async def get_item(id: int):  # BUG: should be item_id, not id
    ...

# ❌ Don't define overlapping routes — FastAPI matches first-registered
@app.get("/items/latest")   # must come BEFORE /items/{item_id}
@app.get("/items/{item_id}")
```

### Dependencies

```python
# ❌ Don't mutate request.state across middleware and route handlers
# State set in middleware may not be visible in dependencies depending on order

# ❌ Don't use mutable default arguments in dependency functions
async def get_settings(config: dict = {}):  # BUG: shared across requests
    ...

# ✅ Use None as default, initialize inside
async def get_settings(config: Optional[dict] = None):
    if config is None:
        config = {}
```

### Background Tasks

```python
# ❌ Don't add background tasks after response has started streaming
# ✅ Add BackgroundTasks as a parameter, don't instantiate manually
async def send_email(background_tasks: BackgroundTasks):
    background_tasks.add_task(email_sender, to="user@example.com")
```

### Testing

```python
# ✅ Use the TestClient as a context manager for lifespan events
with TestClient(app) as client:
    response = client.get("/")

# ❌ Don't instantiate TestClient outside a context manager if your app
#    uses startup/shutdown lifespan events — they won't fire
```

### OpenAPI / Schema

- Adding `response_model=None` silently disables response validation — intentional but easy to misuse
- `response_model_exclude_unset=True` is commonly needed but easy to forget — check the PR if schema looks wrong
- `include_in_schema=False` hides a route from docs but does NOT protect it — don't use it as a security mechanism

## §6 File Change Risk Map

### High Risk — change carefully, full test suite required

| File | Why |
|------|-----|
| `fastapi/routing.py` | Core route registration; high historical churn; breaks propagate everywhere |
| `fastapi/_compat.py` | Pydantic v1/v2 boundary; subtle breakage across Python versions |
| `fastapi/dependencies/utils.py` | Dependency resolution graph; async/sync correctness is non-obvious |
| `fastapi/security/` | Auth logic; security bugs have high severity |

### Medium Risk — test the affected subsystem

| File | Why |
|------|-----|
| `fastapi/encoders.py` | Used by almost everything for serialization; edge cases are plentiful |
| `fastapi/openapi/utils.py` | Schema generation is complex; regressions may not show up in unit tests |
| `fastapi/exception_handlers.py` | Changes affect all error responses globally |

### Low Risk — relatively stable

| File | Why |
|------|-----|
| `fastapi/testclient.py` | Thin re-export of Starlette's TestClient; rarely changes |
| `fastapi/middleware/` | Mostly pass-through wrappers |
| `fastapi/openapi/docs.py` | Static HTML/JS for Swagger UI; cosmetic only |

### Do Not Touch Without Discussion

- `fastapi/_compat.py` — requires coordinated Pydantic v1 + v2 testing
- Any change that removes or renames a public parameter in `params.py` — breaking change

## §7 Testing Conventions

```python
# Fixture pattern used throughout the test suite
from fastapi.testclient import TestClient
import pytest

@pytest.fixture
def client():
    app = create_test_app()
    with TestClient(app) as c:
        yield c

# Test file naming: test_<feature>.py
# Test function naming: test_<behavior>_<condition>
# Example: test_get_item_returns_404_when_not_found
```

- Tests live in `tests/` flat structure — no subdirectory nesting
- Each feature area has its own test file (`test_routing.py`, `test_dependencies.py`, etc.)
- Parametrize aggressively rather than duplicating test functions
- Use `pytest.mark.anyio` for async tests (not `asyncio.run`)
